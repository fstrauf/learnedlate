A comprehensive SAP Clean Core assessment evaluates your system readiness for S/4HANA migration by identifying technical debt, process gaps, and governance issues. It covers custom code analysis, data quality checks, process standardisation, and compliance criteria to help prevent project delays. This framework has been refined through multiple enterprise transformations and is designed to give programme managers, enterprise architects, and SAP basis teams a shared language for readiness.

In this article, I'll share a practical framework I've developed through years of leading S/4HANA transformations. Whether you're preparing for a greenfield implementation, brownfield conversion, or selective data transition, this checklist will help you establish a solid foundation.

## What Is a Clean Core Assessment?

A Clean Core assessment evaluates your existing SAP landscape against the principles of the Clean Core approach. SAP introduced the Clean Core philosophy to help organisations maintain a stable, upgradable system core while enabling innovation through side-by-side extensions and cloud services. If you're new to the concept, start with [SAP Clean Core: What It Is, Why It Matters, and How to Achieve It](/blog/sap_clean_core/) for a thorough grounding before diving into this checklist. Understanding the philosophy behind Clean Core will make each section of this checklist more actionable.

The assessment examines three critical dimensions:

- **Technical cleanliness**: Code quality, modification levels, and technical debt
- **Process standardisation**: Alignment with SAP standard processes versus custom workflows
- **Governance maturity**: Change management, documentation practices, and compliance frameworks

Without this baseline understanding, migration projects often encounter scope creep, budget overruns, and delayed go-live dates. Organisations that skip a formal assessment typically discover critical gaps during the realise phase, when remediation is exponentially more expensive.

## The Clean Core Assessment Checklist

### Technical Assessment

#### 1. Custom Code Analysis

- [ ] Inventory all custom Z* and Y* objects in your ECC system
- [ ] Identify obsolete customisations no longer in use
- [ ] Document all enhancements (user exits, BAdIs, BTEs)
- [ ] Review modifications to SAP standard objects
- [ ] Assess compatibility of custom code with S/4HANA using ATC (ABAP Test Cockpit)
- [ ] Identify custom code requiring remediation for SAP HANA database

**Red flag indicators**: More than 20% of your transaction volume runs through custom code, or you have modifications to core SAP tables like BKPF or BSEG.

#### 2. Data Quality and Volume

- [ ] Analyse database growth trends over the past 3-5 years
- [ ] Identify archiving opportunities for application logs, IDocs, and change documents
- [ ] Assess master data quality (customer, vendor, material, GL accounts)
- [ ] Document data retention requirements by legal entity
- [ ] Evaluate the need for data migration versus data archiving strategies

#### 3. Interface Landscape

- [ ] Map all RFC connections and their purposes
- [ ] Inventory PI/PO interfaces and assess migration to SAP Integration Suite
- [ ] Document IDoc types and message flows
- [ ] Identify file-based integrations requiring modernisation
- [ ] Assess API readiness for cloud connectivity

### Process Assessment

#### 4. Business Process Documentation

- [ ] Document current state processes for all key business areas (OTC, PTP, RTR)
- [ ] Identify process variations across business units or geographies
- [ ] Map process deviations from SAP standard functionality
- [ ] Document regulatory or industry-specific requirements
- [ ] Assess shared service centre implications

#### 5. Organisational Readiness

- [ ] Evaluate current change management capabilities
- [ ] Assess user training requirements and delivery methods
- [ ] Document key user and super-user structures
- [ ] Identify change champions within business units
- [ ] Review governance for future process changes

### Governance Assessment

#### 6. System Landscape Strategy

- [ ] Define target landscape (single tenant vs. multi-tenant)
- [ ] Plan sandbox, development, quality, and production environments
- [ ] Assess SAP BTP requirements for extensions and integrations
- [ ] Review integration architecture against modern API-first patterns. For a broader view on connecting systems, see [API Integration Services: Connecting Your Business Systems Seamlessly](/blog/api_integration_services_connecting_your_business_systems_seamlessly/)
- [ ] Document release and transport management approach
- [ ] Plan for continuous innovation and quarterly upgrades

#### 7. Extension Strategy

- [ ] Categorise all current customisations by business criticality
- [ ] Identify candidates for side-by-side extensions on SAP BTP
- [ ] Plan for SAP Fiori app development or adoption
- [ ] Document in-app extensibility requirements
- [ ] Assess third-party add-ons and their S/4HANA compatibility

For organisations planning to move custom logic off the core, [SAP BTP Side-by-Side Extension: Complete Working Example with Code](/blog/sap_btp_side_by_side_extension_complete_working_example_with_code/) shows exactly how to structure a BTP extension project end to end.

#### 8. AI and Automation Readiness

As S/4HANA deployments increasingly leverage embedded AI features, it is worth assessing your readiness for intelligent automation as part of the Clean Core exercise:

- [ ] Review current RPA and workflow automation footprint
- [ ] Identify processes that could benefit from SAP's AI capabilities
- [ ] Assess SAP Joule eligibility based on your licensing and cloud deployment model
- [ ] Document data quality requirements for AI-assisted process optimisation
- [ ] Evaluate readiness to adopt SAP Build Process Automation

If you plan to extend your S/4HANA investment with AI-assisted capabilities, [SAP Joule Custom Skills Development: Step-by-Step Tutorial for Beginners](/blog/sap_joule_custom_skills_development_step_by_step_tutorial_for_beginners/) explains how to build custom Joule skills on top of BTP.

## Scoring Your Assessment

Once you've worked through the checklist, score each section on a scale of 1-5:

| Score | Interpretation | Recommended Action |
|-------|----------------|-------------------|
| 1-2 | High complexity/risk | Prioritise remediation before migration |
| 3 | Moderate complexity | Plan for remediation during migration |
| 4-5 | Clean and ready | Proceed with migration planning |

A total score below 20 suggests significant preparation work is needed. Scores between 20-30 indicate moderate readiness with some targeted remediation required. Above 30, you're in good shape to proceed with migration planning. If your score is low, don't panic—many organisations start here. The key is to use the results to build a prioritised roadmap rather than attempting to fix everything at once.

## Common Pitfalls to Avoid

Based on my experience with S/4HANA transformations, here are the most common mistakes organisations make during Clean Core assessments:

**Underestimating data quality issues**: Poor master data doesn't magically improve during migration. Start data cleansing early, ideally 12-18 months before go-live.

**Ignoring user adoption**: Technical readiness means nothing if users resist the new system. Include change impact assessments in your evaluation.

**Overlooking integration complexity**: Point-to-point integrations are the silent killers of S/4HANA projects. Modernise your integration architecture before migration.

**Failing to document business rationale**: Every customisation should have a documented business justification. If you can't explain why it exists, it probably shouldn't.

**Neglecting cloud readiness**: S/4HANA is increasingly deployed in cloud or hybrid models. Assess your infrastructure, network latency, and security posture early. ## Building Your Clean Core Roadmap

The assessment is just the beginning. Use your findings to build a phased roadmap:

1. **Quick wins** (0-3 months): Remove unused custom code, archive obsolete data, retire redundant interfaces
2. **Remediation phase** (3-12 months): Refactor critical custom code, standardise processes, implement data quality frameworks
3. **Migration preparation** (12-18 months): Finalise extension strategy, complete user training, execute cutover planning

## Why Clean Core Matters for Long-Term SAP Value

Organisations that treat Clean Core as a one-off pre-migration exercise often find themselves in the same technical debt trap within a few years. The real value of this assessment framework lies in its reuse: quarterly health checks, pre-upgrade validations, and integration governance reviews all benefit from the same structured approach.

A well-maintained Clean Core posture also accelerates adoption of SAP's innovation roadmap. Features like embedded analytics, AI-assisted process recommendations, and industry cloud solutions all assume a standardised core. The more custom code and process deviation you carry, the longer it takes to activate these capabilities—and the higher your total cost of ownership becomes.

For SAP teams, this means shifting from a project mindset to a product mindset. The Clean Core assessment is not a gate before migration; it is a recurring practice that protects your platform investment over its full lifecycle.

## How to Use This Checklist in Your Organisation

To get the most value from this framework, treat the assessment as a collaborative exercise rather than a solo audit. Involve representatives from your SAP basis team, functional leads, enterprise architecture, and business process owners. Each group brings a different perspective on what constitutes "clean" versus "custom," and their combined input produces a more accurate readiness score.

Begin by scheduling a series of workshops, one for each assessment dimension: technical cleanliness, process standardisation, and governance maturity. Allocate half a day per workshop and require participants to bring evidence: system extracts, process documentation, change logs, and integration inventories. The checklist items above serve as discussion prompts, but the real value comes from the conversations they spark.

Document your findings in a shared repository—ideally a living document that the programme office updates as remediation work progresses. Many organisations create a simple dashboard showing red, amber, and green status for each checklist item. This visibility keeps leadership informed and ensures that blockers receive attention before they derail the migration timeline.

Finally, revisit the assessment at key milestones: after system conversion, before user acceptance testing, and prior to go-live. The scores will shift as you retire technical debt and standardise processes, and tracking that progress builds confidence among stakeholders who may not understand the technical details but need assurance that the programme is on track.

A thorough **SAP Clean Core assessment** isn't a box-ticking exercise—it's the foundation of a successful S/4HANA migration. By systematically evaluating your technical landscape, business processes, and governance frameworks, you can identify risks early and build a realistic migration plan.
